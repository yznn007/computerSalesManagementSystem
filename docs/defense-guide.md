# 电脑销售管理系统 — 答辩指南

> 本文档用通俗语言讲解系统全貌，适合不熟悉前后端开发的同学在答辩前快速理解项目。

---

## 1. 这个系统是做什么的？

一句话：**一个在线电脑商城，顾客可以浏览商品并下单，店员可以管理商品、客户和订单。**

系统里有两种角色：

| 角色 | 能做什么 | 类比 |
|------|---------|------|
| **客户（Customer）** | 注册账号 → 浏览商品 → 加入购物车 → 下单 → 付款 → 取消订单 | 就像你在淘宝买东西 |
| **销售员（Staff）** | 管理商品（上架/下架/改价）、管理客户信息、处理订单（发货/退款） | 就像淘宝店家的后台管理员 |

## 2. 系统是怎么跑起来的？（三层架构）

任何一个 Web 系统都可以分成三层，你可以把每层想象成开一家实体店的不同部分：

```
┌─────────────────────────────────────────────────────────┐
│                    前端（Vue 3）                          │
│              = 店面的橱窗和收银台                          │
│    顾客看到的网页，包含商品列表、购物车、订单页面             │
│    运行在你的浏览器里（http://localhost:5173）              │
└──────────────────────┬──────────────────────────────────┘
                       │  HTTP 请求（Ajax）
                       │  "我要下单！"  "有哪些商品？"
                       ▼
┌─────────────────────────────────────────────────────────┐
│                    后端（Spring Boot）                     │
│              = 店里的营业员和规章制度                       │
│    接收前端的请求，判断你有没有权限，然后操作数据库            │
│    运行在服务器上（http://localhost:8080）                  │
└──────────────────────┬──────────────────────────────────┘
                       │  SQL 语句
                       │  SELECT / INSERT / UPDATE
                       ▼
┌─────────────────────────────────────────────────────────┐
│                   数据库（MySQL）                          │
│              = 店里的账本和仓库                            │
│    存储所有数据：商品信息、客户资料、订单记录                 │
│    运行在 MySQL 服务里（端口 3306）                         │
└─────────────────────────────────────────────────────────┘
```

**一条完整的请求是怎么走的？** 举例：顾客点击"下单"按钮：

1. 前端（浏览器）发起 HTTP 请求 → `POST /api/orders`，携带购物车数据
2. 后端（Spring Boot）收到请求 → 校验 JWT token（你是谁）→ 校验权限（你能下单吗）→ 调用存储过程
3. 数据库（MySQL）执行存储过程 → 检查库存 → 扣减库存 → 生成订单号 → 返回结果
4. 后端把结果包装成 JSON → 返回给前端
5. 前端显示"下单成功，订单号：xxx"

## 3. 为什么选这些技术？（技术选型理由）

| 技术 | 选它做什么 | 为什么选它 |
|------|----------|-----------|
| **Vue 3** | 前端框架 | 中文文档完善、学习曲线平缓、国内企业使用广泛；组合式 API 让代码更好组织 |
| **Element Plus** | UI 组件库 | 饿了么团队出品，提供现成的表格/表单/弹窗/抽屉等组件，不用从零写样式 |
| **Vite** | 构建工具 | 新一代前端构建工具，启动快（冷启动秒级）、热更新即时生效 |
| **Spring Boot** | 后端框架 | Java 生态最主流的 Web 框架，自动配置、内嵌服务器（不用部署 Tomcat） |
| **MyBatis** | 数据库访问 | 比 Hibernate 更轻量，SQL 完全可控；注解模式无需 XML 配置文件 |
| **MySQL** | 数据库 | 开源免费、性能稳定、支持事务（InnoDB 引擎）、中文支持好 |
| **JWT** | 身份认证 | 无状态 token，服务端不存 session，天然支持分布式；Base64 编码可携带用户信息 |
| **BCrypt** | 密码加密 | 自带盐值的哈希算法，同样的密码每次哈希结果不同，防止彩虹表攻击 |

## 4. 数据库是怎么设计的？（9 张表）

数据库设计的核心思想是 **「一张表管一件事」**，这叫数据库范式（3NF）。

### 4.1 核心表说明

```
            ┌──────────┐         ┌──────────┐
            │ Customer │         │  Staff   │    ← 账号表（双角色）
            │  客户表   │         │ 销售员表  │
            └─────┬─────┘         └──────────┘
                  │ 1
                  │ 下单
                  │ N
            ┌─────┴──────┐
            │ Sales_Order │                         ← 订单主表
            │   订单表     │
            └─────┬──────┘
                  │ 1
                  │ 包含
                  │ N
            ┌─────┴──────┐      ┌──────────┐
            │Order_Detail │─────→│ Product  │       ← 订单明细 + 商品表
            │  订单明细    │  N:1 │  商品表   │
            └─────────────┘      └────┬─────┘
                                      │ 1
                     ┌────────────────┼────────────────┐
                     │ 1              1                 │ N
               ┌─────┴─────┐  ┌──────┴──────┐  ┌──────┴──────┐
               │  Laptop   │  │  Desktop    │  │ Spare_Part  │  ← 按品类垂直拆分的详情表
               │  笔记本详情 │  │  台式机详情  │  │  配件详情    │
               └───────────┘  └──────┬──────┘  └──────┬──────┘
                                     │ 1               │ N
                                     │    组装配置      │
                                     └──→ Desktop_Composition ←┘
```

### 4.2 为什么要这样设计？

- **商品表（Product）存公共字段**（品牌、型号、价格、库存），**详情表按品类拆分**（笔记本有屏幕尺寸，台式机有机箱类型，配件有规格参数）。这样不会出现「笔记本有硬盘转速字段」「配件有屏幕尺寸字段」这种奇怪的空列。
- **订单主表 + 订单明细表**（一对多）：一笔订单可以买多件不同商品，每件商品的数量、单价单独记录。
- **台式机组装配置表**：记录一台台式机用了哪些 DIY 配件（比如 CPU、显卡、内存各一个），这样客户买台式机时可以看到完整的配置清单。

## 5. 核心业务流程

### 5.1 客户注册与登录

```
客户打开网页 → 点击"注册" → 填写姓名/手机号/地址/密码
→ 后端校验手机号是否唯一 → BCrypt 加密密码 → 存入 Customer 表 → 注册成功

客户登录 → 输入手机号+密码 → 后端查询 Customer 表 → BCrypt 校验密码
→ 生成 JWT token（含用户ID、角色、姓名）→ 返回 token 给前端
→ 前端存到 localStorage → 后续每次请求自动携带 token
```

### 5.2 浏览商品与下单

```
客户进入主页 → 看到商品列表（可按品类/关键词筛选，按名称/价格/库存排序）
→ 点击"+"加入购物车 → 打开购物车抽屉查看 → 调整数量 → 点击"提交订单"
→ 前端发送 {items: [{product_id:1, quantity:2}, ...]}
→ 后端调用存储过程 sp_create_order：
   ① 开启事务
   ② SELECT ... FOR UPDATE 锁定要买的商品行（防止别人同时买导致超卖）
   ③ 检查库存是否足够
   ④ 扣减库存
   ⑤ 生成订单号（ORD+时间戳+随机数）
   ⑥ 写入 Sales_Order + Order_Detail
   ⑦ 提交事务
→ 返回订单号给前端 → 前端弹窗"下单成功"
```

### 5.3 订单生命周期（状态流转）

```
  ┌─────────┐   付款(pay)   ┌─────────┐   发货(ship)   ┌─────────┐
  │  待付款  │ ────────────→ │  已付款  │ ────────────→ │  已发货  │
  └────┬────┘               └─────────┘               └────┬────┘
       │ 取消(cancel)                                       │ 退货(return)
       ▼                                                    ▼
  ┌─────────┐                                         ┌─────────┐
  │  已取消  │                                         │  已退货  │
  └─────────┘                                         └─────────┘
  （库存回补）                                          （库存回补）
```

**权限控制**（谁能在什么时候做什么操作）：
- 客户可以对自己的订单 **付款** 和 **取消**
- 销售员可以对任意订单 **发货**、**取消**、**退货**，但 **不能替客户付款**
- 取消和退货时，系统会自动把商品库存加回去

## 6. 安全设计（答辩加分项）

### 6.1 密码安全 — BCrypt 加密

**问题**：如果数据库被盗，攻击者能直接看到用户密码吗？

**方案**：密码从不存明文，而是存 BCrypt 哈希值。

```
用户输入 "123456" → BCrypt("123456") → "$2a$10$N9qo8uLOickgx2ZMRZoMye..."
                                              ↑
                                    每次哈希结果都不同（自带随机盐值）
```

登录时：拿用户输入的密码，用 BCrypt 的 `matches()` 方法和数据库里的哈希值比对，无需知道原始密码。

### 6.2 身份认证 — JWT（JSON Web Token）

**问题**：HTTP 是无状态协议，服务器怎么知道"你是谁"？

**方案**：登录成功后，后端发一个「电子身份证」（JWT token）给前端。后续每次请求，前端把这个 token 放在请求头里。

```
登录成功 → 后端生成 JWT：
  Header:  {"alg": "HS256"}                     ← 签名算法
  Payload: {"sub": 1, "role": "customer", ...}   ← 用户信息
  Signature: HMAC-SHA256(Header+Payload, 密钥)    ← 防篡改签名

后续请求 → 前端带上：Authorization: Bearer <token>
        → 后端验证签名 → 提取用户信息 → 写入 ThreadLocal（当前请求的临时存储）
        → 控制器通过 AuthContext.require() 获取当前用户
```

**为什么用 JWT 而不是 Session？**
- JWT 是无状态的，服务端不用存登录信息，天然支持集群部署（多台服务器共用同一个密钥即可）
- token 自带用户信息，不用每次都查数据库

### 6.3 权限控制 — 基于角色的访问控制

```
请求进来 → JwtAuthFilter 解析 token → AuthContext.set(用户)
→ Controller 调用 requireStaff() 检查角色
→ 如果不是 staff → 抛出 BizException(403, "需要销售员权限")
```

关键权限规则（容易搞错的地方）：
- **客户下单**：customer_id 强制取 token 里的，不是请求体传的（防伪造）
- **销售员代客下单**：才使用请求体里的 customer_id

### 6.4 业务安全 — 悲观锁防超卖

**问题**：如果 A 和 B 同时买最后一件商品，会不会超卖？

```
时间线（没有锁的情况）：
  用户A: 查库存 → 库存=1       用户B: 查库存 → 库存=1
  用户A: 下单，库存改为 0       用户B: 下单，库存改为 0  ← 超卖了！
```

**方案**：下单时用 `SELECT ... FOR UPDATE`（悲观锁），谁先查到谁先锁住，另一个人必须等。

```
时间线（有锁的情况）：
  用户A: SELECT ... FOR UPDATE → 锁住商品行
  用户B: SELECT ... FOR UPDATE → 等待...（A 还没释放锁）
  用户A: 扣库存 → 提交事务 → 释放锁
  用户B: 查到库存=0 → 返回"库存不足"
```

## 7. 前后端如何通信？（API 接口）

前后端通过 **RESTful API** 通信，简单说就是前端发 HTTP 请求，后端返回 JSON 数据。

### 7.1 接口约定

| 约定 | 说明 |
|------|------|
| 请求路径 | 统一以 `/api` 开头，如 `/api/products`、`/api/orders` |
| 请求格式 | JSON（Content-Type: application/json） |
| 响应格式 | JSON，字段名用下划线（`product_id`、`order_no`） |
| 鉴权方式 | 请求头带 `Authorization: Bearer <token>` |
| 错误格式 | `{"code": 400, "message": "错误描述"}` |

### 7.2 主要接口一览

| 功能 | 方法 | 路径 | 说明 |
|------|------|------|------|
| 客户注册 | POST | `/api/auth/register` | 公开接口 |
| 客户登录 | POST | `/api/auth/login/customer` | 公开接口 |
| 销售员登录 | POST | `/api/auth/login/staff` | 公开接口 |
| 查看个人信息 | GET | `/api/auth/me` | 需要登录 |
| 修改密码 | PUT | `/api/auth/me/password` | 需要登录 |
| 商品列表 | GET | `/api/products` | 需要登录，可按品类筛选 |
| 商品详情 | GET | `/api/products/{id}` | 需要登录 |
| 新增商品 | POST | `/api/products` | 仅销售员 |
| 编辑商品 | PUT | `/api/products/{id}` | 仅销售员 |
| 删除商品 | DELETE | `/api/products/{id}` | 仅销售员 |
| 客户列表 | GET | `/api/customers` | 仅销售员 |
| 新增客户 | POST | `/api/customers` | 仅销售员 |
| 下单 | POST | `/api/orders` | 需要登录 |
| 订单列表 | GET | `/api/orders` | 需要登录（客户只看自己的） |
| 订单详情 | GET | `/api/orders/{id}` | 需要登录 |
| 状态流转 | PUT | `/api/orders/{id}/status` | 需要登录（按角色限制操作） |

## 9. 答辩常见问题（Q&A）

### Q1：你这个系统有什么实际价值？
A：模拟了一个真实的电脑商城。支持三种商品类型（笔记本、台式机、DIY 配件），台式机可以配置组装清单（CPU + 显卡 + 内存等）。双角色设计区分了买家和卖家，有完整的订单状态流转。

### Q2：订单状态机是怎么回事？
A：订单有 5 个状态（待付款→已付款→已发货→已退货，以及已取消）。每个状态只能进行特定的操作（比如待付款才能付款、已付款才能发货）。这个状态流转逻辑封装在数据库存储过程中，保证数据一致性。

### Q3：怎么防止商品超卖？
A：使用数据库的悲观锁（`SELECT ... FOR UPDATE`）。下单时先锁定要买的商品行，其他并发请求必须等待，锁释放后重新读库存，发现不够就会拒绝下单。

### Q4：密码安全怎么保证？
A：使用 BCrypt 单向哈希算法。密码从不存明文，数据库里存的是一串哈希值。BCrypt 自带随机盐值，即使两个用户密码相同，哈希值也不同，有效防止彩虹表攻击。

### Q5：JWT 是什么？怎么用的？
A：JWT（JSON Web Token）是一个加密签名的字符串，里面可以安全地携带用户 ID、角色等信息。服务端不需要存登录状态，只需验证 token 的签名是否合法。每次请求前端自动带上 token，后端解析后就知道是谁在操作。

### Q6：项目有什么可以改进的地方？
A：可以增加销售报表统计（SalesReport.vue 目前是占位页），引入 Redis 实现购物车持久化，增加商品图片上传，增加订单物流跟踪，引入单元测试覆盖核心业务逻辑等。

## 10. 答辩演示步骤建议

### 准备阶段
1. 确保 MySQL 服务已启动，且执行了全部 5 个 SQL 脚本
2. 启动后端：`.\SpringBoot\mvnw.cmd -f .\SpringBoot\pom.xml spring-boot:run`
3. 启动前端：`cd vue3 && npm run dev`
4. 浏览器打开 `http://localhost:5173`

### 演示流程（约 5-8 分钟）

**Step 1：展示登录页**（30秒）
- 展示双角色登录 tab（客户登录 / 销售员登录）
- 展示注册弹窗

**Step 2：客户视角演示**（2分钟）
- 用 `13800138001` / `123456` 登录
- 浏览商品，切换分类（笔记本/台式机/DIY配件）
- 搜索 + 排序

- 取消一笔待付款订单

**Step 3：销售员视角演示**（2分钟）
- 用 `admin` / `admin` 登录

**Step 4：技术亮点讲解**（1分钟）

- 讲解三层架构（前端→后端→数据库）
